## この記事について

この記事は、OpenClaw のワークスペースを**安全に共有する**ための実務向けガイドです。元データは非常に巨大（100万文字超）だったため、運用に必要な情報だけを約1万文字に圧縮しています。機密情報（token / secret / api key / webhook URL など）は掲載せず、構造理解・運用設計・再現手順に絞ってまとめています。

---

## 1. ワークスペースの全体構造（役割ベース）

OpenClaw の運用は、ファイルを「用途」で分けると理解しやすくなります。

```text
workspace/
├─ AGENTS.md              # 運用ルール（毎セッションの動作規約）
├─ SOUL.md                # 口調・人格・境界
├─ USER.md                # ユーザー文脈（呼称/プロジェクト）
├─ MEMORY.md              # 長期記憶（主にmain session）
├─ memory/YYYY-MM-DD.md   # 日次ログ（短期記憶）
├─ HEARTBEAT.md           # 定期チェックの指示書
├─ TOOLS.md               # ローカル環境メモ
├─ skills/                # カスタムスキル群
├─ articles/              # Zenn記事原稿
├─ app/, lib/, components/# 実装コード
└─ .env.local             # 機密情報（外部共有禁止）
```

この分割を守ると、以下の3つが安定します。

1. 迷子にならない（どこに何を書くか明確）
2. セッション再開が速い（読む順が決まっている）
3. 機密漏えいが減る（共有対象を切り分けやすい）

---

## 2. 主要MDファイルの役割

### AGENTS.md
- 「毎セッション最初に読むもの」
- 読み込み順（SOUL / USER / memory / MEMORY）
- グループチャットでの発話ルール
- 外部アクション時の確認方針

### SOUL.md
- 回答トーン（端的、実務優先、媚びない 等）
- 行動原則（まず調べる、秘密を漏らさない）
- 境界条件（迷う外部行動は確認）

### USER.md
- ユーザー固有の文脈を保持
- 開発中プロジェクト、関心テーマ、運用方針
- これがあると提案の精度が上がる

### MEMORY.md / memory/*.md
- MEMORY.md: 長期記憶（厳選）
- memory/日次: 当日の出来事やログ
- 心得: 「覚えたいことはファイルへ書く」

### HEARTBEAT.md
- 定期ポーリング時に何を確認するかを制御
- 空なら無駄なAPIコールを抑制
- Cronとの住み分けが重要

## 3. 「共有してよい情報」と「共有禁止情報」

### 共有してよい
- ディレクトリ構造
- 運用ルール（AGENTS/SOULの方針）
- スキルの目的・使い分け
- ワークフロー図、ジョブ設計、KPI

### 共有禁止（必ずマスク）
- Discord/Slack/Telegram などの token
- gateway auth token
- OPENAI_API_KEY などの API key
- webhook URL / session key / access secret
- 個人情報につながるIDの生値

実務では、設定ファイルそのものを貼るのではなく、**マスク済みサンプル**で共有します。

## 5. Skills一覧（公開可能な範囲）

このワークスペースでは、少なくとも以下のカスタムSkillを運用しています。

1. `startups-rip-pathfinder`
   - 新規プロダクト企画前に startups.rip で類似失敗を10件以上調査
   - 失敗理由をパターン化し、失敗回避策に変換

2. `gemini-video-understanding`
   - 動画を分割し、転写/要約/Embedding/RAG検索に接続
   - 「動画理解を再現可能手順にする」ための設計スキル

3. `chrome-extension-generator`
   - Chrome拡張のテンプレ作成と初期実装を高速化

4. `twitter-x-api`
   - X API操作の実務補助（認証前提）

5. `x-jp-translator-builder`
   - X投稿の本文翻訳ワークフローを拡張開発用に整理

## 6. SOUL.md を入れる理由（重要）

`SOUL.md` は単なる自己紹介ではなく、**出力品質の固定装置**です。

- セッションが変わっても回答スタイルがぶれない
- 方針（実務優先・安全優先・プライバシー重視）を維持できる
- チーム運用時の人格ドリフトを防げる

「毎回同じ品質で答える」ために、実は最重要ファイルのひとつです。

---

## 7. サブエージェント/ACP運用での注意

- ACP実行環境が落ちるケースがあるため、代替フロー（直接実装）を準備
- 長時間タスクは進捗チャンネルへ逐次ログ化
- 失敗理由（権限/認証/依存不足）を必ず残す

運用ルールとしては、

- できるだけ再現可能に
- 成功より失敗ログを重視
- 修正をスキルへ還元

この3点を守ると改善速度が上がります。

---

## 8. 記事運用（Zenn）でハマりやすい点

- 超巨大記事はプレビューが不安定になりやすい
- Frontmatter崩れ（`---`の壊れ）でパースエラーが出る
- パス区切りやコードフェンス未閉じで崩壊しやすい

対策:

1. 1記事は1万文字前後に分割
2. frontmatterを最初に固定
3. code blockの開閉を自動検査
4. `published: false` で確認後に公開

---

## 9. 最低限の運用テンプレ

### 日次ループ
- 朝: HEARTBEAT/優先タスク確認
- 昼: 実装・調査・ログ更新
- 夜: memory日次更新 + 失敗学習のルール化

### 共有ループ
- 共有前にマスク
- 共有後に再チェック（token混入確認）
- 問題あれば即ローテーション

## 10. まとめ

OpenClaw運用で大事なのは、

- 構造を役割で分ける
- ルールをファイル化して再現性を持たせる
- 機密を絶対に分離する

です。

巨大ログをそのまま公開するより、
**構造・目的・手順・安全基準**を明文化したほうが、チーム全体の実行力は確実に上がります。

- [ ] code block が閉じている

このチェックを通せば、ほぼ事故なく公開できます。

## 11. 実運用で効いた改善パターン（重複なし版）

### 11.1 ログ設計

運用ログは「量」より「構造」が重要です。最低限、次の5項目を固定すると再現性が上がります。

- `job_name`: 実行ジョブ名
- `started_at` / `ended_at`: 実行時間
- `status`: success / failed / partial
- `error_type`: API / auth / rate_limit / validation
- `artifact`: 生成物の参照先（URL/ID）

この形式を統一すると、障害分析・自動集計・改善提案が機械的に回ります。

### 11.2 承認フロー

SNS運用や外部投稿系は、初期は必ず承認付きにするのが安全です。推奨ステート:

`Draft -> Review -> Approved -> Published -> Evaluated`

- 高リスク投稿（炎上可能性、事実確認が必要な内容）は `Review` を必須化
- 低リスク投稿（定型告知）は条件付き自動化

### 11.3 失敗知識の昇格

失敗は「ログに残す」だけだと再発します。次の昇格ルールを入れると効果的です。

- 同じ失敗が2回出たら `つまずきと学び` に記録
- 3回目が起きる前に `再現ガイド` に回避手順を追記
- 4回目を防ぐため `skills` の手順へ反映

## 12. Skills運用の設計指針

### 12.1 Skillを作るべきタイミング

以下の条件を2つ満たしたらSkill化を検討します。

- 同じ作業を3回以上繰り返している
- 手順ミスが発生しやすい
- 依存情報（参照URL/順序）が多い
- 品質を固定したい

### 12.2 SKILL.mdの最小構成

- `name`
- `description`（トリガーワードを含める）
- `goal`
- `workflow`（番号付き）
- `output format`
- `quality rules`

descriptionが曖昧だとスキル選定精度が落ちるため、用途とトリガー語を明示します。

### 12.3 代表的なワークスペース内Skill

- `startups-rip-pathfinder`: 失敗事例を先に収集し、企画の地雷回避へ変換
- `gemini-video-understanding`: 動画RAGの手順を規格化
- `x-jp-translator-builder`: X投稿翻訳拡張の実装補助

## 13. セキュリティ運用の実務ルール

### 13.1 公開前スキャン

記事やREADMEを公開する前に、最低でも以下を検索してヒット有無を確認します。

- `token`
- `api_key`
- `secret`
- `authorization`
- `webhook`
- `session`

ヒットしたら即公開停止、マスク後に再確認。

### 13.2 秘密情報の分離

- `openclaw.json`: 運用設定（機密を含む）
- `.env.local`: 秘密値（絶対に公開しない）
- 公開資料: マスク済みサンプルのみ

### 13.3 事故時対応

誤って公開した場合の優先順位:

1. 公開停止（記事/コミットを即非公開）
2. 対象トークンをローテーション
3. 影響範囲をログから特定
4. 再発防止ルールを追記

## 14. Zenn連携時の実践チェック

- 記事が極端に長い場合は分割（推奨）
- frontmatterの整合性チェック
- code block未閉じチェック
- 画像/リンク切れ確認
- 公開後の表示崩れ確認

この順番で確認すると、公開トラブルが大幅に減ります。

## 15. 最終まとめ

このワークスペース運用で重要なのは次の4点です。

1. **構造を役割で分離**して迷いをなくす
2. **SOUL/AGENTSで出力品質を固定**する
3. **Skill化で再現性を上げる**
4. **機密情報を徹底分離**する

この4つを守るだけで、個人運用でもチーム運用でも品質が安定します。

## 16. 運用Q&A（よく詰まる点）

**Q. memory系はどこまで読むべき？**  
A. main sessionは `MEMORY.md + 当日/前日daily`、共有チャットはdaily中心が安全です。

**Q. スキルが増えると管理しづらい**  
A. descriptionにトリガー語を入れ、似た用途は統合、使われないskillはarchiveへ移します。

**Q. ログが増えすぎる**  
A. ログは「成功詳細」より「失敗詳細」に寄せると、運用価値が上がります。

**Q. 大きい記事は公開してよい？**  
A. 可能ですが、可読性と保守性の観点で章分割が推奨です。

## 17. 実装・調査・公開の標準手順（テンプレ）

1. 要件を1段落で固定する
2. 調査ソースを列挙する
3. 実装前に成功条件を定義
4. 変更を最小単位でcommit
5. ログチャネルへ進捗を投稿
6. 公開前に機密スキャン
7. 公開後に結果を振り返り

この手順を繰り返すことで、プロジェクト横断でも品質が揃います。

## 18. 参考: 共有用ワークスペース棚卸しフォーマット

以下のテンプレを使うと、公開可能/非公開を機械的に分類できます。

```text
[カテゴリ] 設定
- ファイル: openclaw.json
- 公開可否: 条件付き（マスク必須）
- 理由: token/secretが含まれる可能性

[カテゴリ] 人格/運用
- ファイル: SOUL.md, AGENTS.md
- 公開可否: ほぼ可
- 理由: 運用知見として有用

[カテゴリ] 記憶
- ファイル: MEMORY.md, memory/*.md
- 公開可否: 原則不可（要要約）
- 理由: 個人情報/意思決定履歴を含む

[カテゴリ] スキル
- ファイル: skills/*/SKILL.md
- 公開可否: 可
- 理由: 再利用可能な手順資産

[カテゴリ] 実装
- ファイル: app/, lib/, components/
- 公開可否: 可（鍵情報除く）
- 理由: コード共有価値が高い
```

## 19. 監査観点（チェック項目）

### 19.1 機密
- 生tokenは含まれていないか
- Webhook URLが露出していないか
- APIキー例が実値になっていないか

### 19.2 再現性
- 手順が番号化されているか
- 依存条件（Node版、CLI）が明記されているか
- 失敗時の戻し方があるか

### 19.3 可読性
- 章立てがあるか
- 目的→手順→注意点→まとめ の順になっているか
- 長文は分割されているか

- 全量ダンプは事故率が高い
- 要約しすぎると再現性が落ちる

**構造化 + マスク + 役割説明 + 実行テンプレ**

- 実値は公開しない（常にプレースホルダ）
- 個人識別情報は省略または匿名化
- URLは必要最小限のみ記載
- 手順は再現性優先で簡潔に
- 問題発生時は即ロールバックして再公開

この5原則を守れば、実運用でも安全にナレッジ公開できます。

## 22. 補足: 運用レベルを上げるための実践メモ

- 同じ作業は必ずテンプレ化し、次回の入力負荷を下げる
- 失敗理由は「人・設定・外部要因」に分けると対策が立てやすい
- 週1回、不要ファイルと古いログの棚卸しを行う
- 設定変更時は理由を1行で残す（未来の自分向け）
- 記事化するときは、手順のスクリーンショットより先に前提条件を書く
- 公開後に質問が来た部分は、FAQとして追記し記事を育てる
- 重要な運用知見はskillsへ昇格し、個人依存を減らす
- 監査や引き継ぎを想定して、ファイル命名規則を崩さない

以上を継続すると、OpenClaw運用は“思いつき運用”から“再現可能運用”へ進化します。
