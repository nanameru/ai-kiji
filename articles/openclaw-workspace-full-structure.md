## この記事について

この記事は、OpenClaw のワークスペースを**安全に共有する**ための実務向けガイドです。元データは非常に巨大（100万文字超）だったため、運用に必要な情報だけを約1万文字に圧縮しています。機密情報（token / secret / api key / webhook URL など）は掲載せず、構造理解・運用設計・再現手順に絞ってまとめています。

---

## 1. ワークスペースの全体構造（役割ベース）

OpenClaw の運用は、ファイルを「用途」で分けると理解しやすくなります。

```text
workspace/
├─ AGENTS.md              # 運用ルール（毎セッションの動作規約）
├─ SOUL.md                # 口調・人格・境界
├─ USER.md                # ユーザー文脈（呼称/プロジェクト）
├─ MEMORY.md              # 長期記憶（主にmain session）
├─ memory/YYYY-MM-DD.md   # 日次ログ（短期記憶）
├─ HEARTBEAT.md           # 定期チェックの指示書
├─ TOOLS.md               # ローカル環境メモ
├─ skills/                # カスタムスキル群
├─ articles/              # Zenn記事原稿
├─ app/, lib/, components/# 実装コード
└─ .env.local             # 機密情報（外部共有禁止）
```

この分割を守ると、以下の3つが安定します。

1. 迷子にならない（どこに何を書くか明確）
2. セッション再開が速い（読む順が決まっている）
3. 機密漏えいが減る（共有対象を切り分けやすい）

---

## 2. 主要MDファイルの役割

### AGENTS.md
- 「毎セッション最初に読むもの」
- 読み込み順（SOUL / USER / memory / MEMORY）
- グループチャットでの発話ルール
- 外部アクション時の確認方針

### SOUL.md
- 回答トーン（端的、実務優先、媚びない 等）
- 行動原則（まず調べる、秘密を漏らさない）
- 境界条件（迷う外部行動は確認）

### USER.md
- ユーザー固有の文脈を保持
- 開発中プロジェクト、関心テーマ、運用方針
- これがあると提案の精度が上がる

### MEMORY.md / memory/*.md
- MEMORY.md: 長期記憶（厳選）
- memory/日次: 当日の出来事やログ
- 心得: 「覚えたいことはファイルへ書く」

### HEARTBEAT.md
- 定期ポーリング時に何を確認するかを制御
- 空なら無駄なAPIコールを抑制
- Cronとの住み分けが重要

---

## 3. 「共有してよい情報」と「共有禁止情報」

### 共有してよい
- ディレクトリ構造
- 運用ルール（AGENTS/SOULの方針）
- スキルの目的・使い分け
- ワークフロー図、ジョブ設計、KPI

### 共有禁止（必ずマスク）
- Discord/Slack/Telegram などの token
- gateway auth token
- OPENAI_API_KEY などの API key
- webhook URL / session key / access secret
- 個人情報につながるIDの生値

実務では、設定ファイルそのものを貼るのではなく、**マスク済みサンプル**で共有します。

## 4. 代表的なマスク済み設定イメージ

```json
{
  "agents": {
    "defaults": {
      "model": { "primary": "openai-codex/gpt-5.3-codex" },
      "workspace": "C:\Users\<user>\.openclaw\workspace"
    }
  },
  "tools": {
    "profile": "full",
    "exec": { "host": "gateway", "security": "full", "ask": "off" }
  },
  "channels": {
    "discord": {
      "enabled": true,
      "token": "***REDACTED***",
      "allowFrom": ["discord:<ALLOWED_USER_ID>"]
    }
  },
  "gateway": {
    "auth": { "mode": "token", "token": "***REDACTED***" }
  }
}
```

## 5. Skills一覧（公開可能な範囲）

このワークスペースでは、少なくとも以下のカスタムSkillを運用しています。

1. `startups-rip-pathfinder`
   - 新規プロダクト企画前に startups.rip で類似失敗を10件以上調査
   - 失敗理由をパターン化し、失敗回避策に変換

2. `gemini-video-understanding`
   - 動画を分割し、転写/要約/Embedding/RAG検索に接続
   - 「動画理解を再現可能手順にする」ための設計スキル

3. `chrome-extension-generator`
   - Chrome拡張のテンプレ作成と初期実装を高速化

4. `twitter-x-api`
   - X API操作の実務補助（認証前提）

5. `x-jp-translator-builder`
   - X投稿の本文翻訳ワークフローを拡張開発用に整理

## 6. SOUL.md を入れる理由（重要）

`SOUL.md` は単なる自己紹介ではなく、**出力品質の固定装置**です。

- セッションが変わっても回答スタイルがぶれない
- 方針（実務優先・安全優先・プライバシー重視）を維持できる
- チーム運用時の人格ドリフトを防げる

「毎回同じ品質で答える」ために、実は最重要ファイルのひとつです。

---

## 7. サブエージェント/ACP運用での注意

- ACP実行環境が落ちるケースがあるため、代替フロー（直接実装）を準備
- 長時間タスクは進捗チャンネルへ逐次ログ化
- 失敗理由（権限/認証/依存不足）を必ず残す

運用ルールとしては、

- できるだけ再現可能に
- 成功より失敗ログを重視
- 修正をスキルへ還元

この3点を守ると改善速度が上がります。

---

## 8. 記事運用（Zenn）でハマりやすい点

- 超巨大記事はプレビューが不安定になりやすい
- Frontmatter崩れ（`---`の壊れ）でパースエラーが出る
- パス区切りやコードフェンス未閉じで崩壊しやすい

対策:

1. 1記事は1万文字前後に分割
2. frontmatterを最初に固定
3. code blockの開閉を自動検査
4. `published: false` で確認後に公開

---

## 9. 最低限の運用テンプレ

### 日次ループ
- 朝: HEARTBEAT/優先タスク確認
- 昼: 実装・調査・ログ更新
- 夜: memory日次更新 + 失敗学習のルール化

### 共有ループ
- 共有前にマスク
- 共有後に再チェック（token混入確認）
- 問題あれば即ローテーション

---

## 10. まとめ

OpenClaw運用で大事なのは、

- 構造を役割で分ける
- ルールをファイル化して再現性を持たせる
- 機密を絶対に分離する

です。

巨大ログをそのまま公開するより、
**構造・目的・手順・安全基準**を明文化したほうが、チーム全体の実行力は確実に上がります。

---

## 付録: 公開時チェックリスト

- [ ] token / api key / secret をマスクした
- [ ] 個人IDや内部URLを伏せた
- [ ] frontmatter が壊れていない
- [ ] code block が閉じている
- [ ] `published: false` で最終確認した

このチェックを通せば、ほぼ事故なく公開できます。
